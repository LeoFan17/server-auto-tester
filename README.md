## 功能

- 采集 CPU 型号、核心数、频率、使用率、系统负载
- 采集内存总量、使用量、可用量、Swap 状态
- 可配置告警阈值，输出告警列表
- 支持人类可读和 JSON 两种输出格式
- **新增**：三档预设压测方案（冒烟 / 标准交付 / 极限烤机）
- **新增**：压测过程中实时采样，峰值/均值自动汇总为终端表格
- **新增**：压测前状态检查 + `--force` 强制覆盖（人有最高选择权）
- 退出码对接 CI（0=正常，1=有告警/超标，2=采集失败，3=条件不满足）

---

## 环境要求

- Python 3.6+
- Linux 系统
- 必备命令：`lscpu`、`free`、`top`、`cat /proc/loadavg`、`cat /proc/meminfo`
- `dmidecode` 可选（需要 root，用于补充 CPU 型号和频率信息）
- `stress` 可选（仅 `stress_test.py` 压测模式需要）

```bash
# 安装 stress
sudo apt install stress          # Debian / Ubuntu
sudo yum install stress          # RHEL / CentOS
```

## 快速开始

```bash
# 查看当前 CPU / 内存状态
python3 cpu_mem_check.py

# JSON 输出（适合 CI 解析）
python3 cpu_mem_check.py --json

```

## cpu_mem_check.py — 状态快照

采集某一时刻的 CPU 和内存信息，对比阈值输出健康报告。

### 用法

```bash
python3 cpu_mem_check.py [选项]
```

| 选项 | 默认值 | 说明 |
|------|--------|------|
| `--json` | 关 | 输出 JSON 格式 |
| `--cpu 80` | 80 | CPU 使用率告警阈值（%） |
| `--mem 85` | 85 | 内存使用率告警阈值（%） |
| `--load 1.5` | 1.5 | 1分钟负载/核心数告警比例 |

### 输出示例

```
──────────────────────────────────────────────────────────────
  CPU / 内存 检查报告
  采集时间: 2025-06-22T10:00:00

  ▸ CPU
    型号          : Intel(R) Xeon(R) Gold 6430
    架构          : x86_64
    物理核心      : 16
    逻辑核心      : 32
    最高频率      : 4000 MHz
    总使用率      : 12.3%
    负载 (1/5/15) : 0.45 / 0.52 / 0.48

  ▸ 内存
    总量          : 62.54 GB
    已用          : 8.21 GB
    可用          : 54.33 GB
    使用率        : 13.1%
    Swap 总量     : 4.00 GB
    Swap 已用     : 0.00 GB
    Swap 使用率   : 0.0%

  ▸ 告警: 无
──────────────────────────────────────────────────────────────
```

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 正常，无告警 |
| 1 | 存在告警 |
| 2 | 采集失败 |

### 告警规则

| 指标 | 条件 | 级别 |
|------|------|------|
| CPU 使用率 ≥ 阈值 | ≥ 95% | CRITICAL |
| | ≥ 阈值但 < 95% | WARNING |
| 1分钟负载 / 核心数 ≥ 负载系数 | ≥ 1.5 | WARNING |
| 内存使用率 ≥ 阈值 | ≥ 95% | CRITICAL |
| | ≥ 阈值但 < 95% | WARNING |
| Swap 使用率 ≥ 50% | ≥ 50% | WARNING |

## stress_test.py — 自动化压测

基于 `stress` 的 CPU / 内存压测工具，支持预设方案、实时采样、结果汇总。

> **依赖**：`stress` 命令 + `cpu_mem_check.py`（同目录）

### 用法

```bash
python3 stress_test.py [选项]
```

不指定 `--mode` 时，脚本进入交互模式，提示用户选择"查看"或"压测"。

### 预设方案

| 方案 | 标签 | CPU 线程 | 内存 | 时长 | 适用场景 |
|------|------|----------|------|------|----------|
| `1` | 快速冒烟 | 2 线程 | 512M | 30s | 上架 / 重启后确认硬件在位 |
| `2` | 标准交付 | 物理核数 | 可用内存 80% | 5m | 给客户的验收报告 |
| `3` | 极限烤机 | 全部逻辑核 | 可用内存 95% | 30m | 上线前稳定性兜底 |

> 方案 2 和 3 的内存值根据当前系统可用内存**动态计算**，不会写死。

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--mode view` | 交互选择 | 仅查看当前状态 |
| `--mode stress` | 交互选择 | 执行压测 |
| `--profile 2` | 2 | 预设方案编号（1 / 2 / 3） |
| `--force` | 关 | 强制压测，跳过状态检查 |
| `--json` | 关 | 输出 JSON 格式 |
| `--cpu 80` | 80 | CPU 告警 & 通过阈值（%） |
| `--mem 85` | 85 | 内存告警 & 通过阈值（%） |
| `--load 1.5` | 1.5 | 负载/核心告警比例 |
| `--stress-cpu 8` | 无 | 覆盖方案中的 CPU 线程数 |
| `--stress-mem 1G` | 无 | 覆盖方案中的内存量（如 `512M` / `1G`） |
| `--stress-time 10m` | 无 | 覆盖方案中的时长（如 `30s` / `5m` / `1h`） |

```

  指标             │      峰值 │      均值 │ 判定
  ────────────────┼──────────┼──────────┼────────────────

  内存使用率       │    81.5% │    78.9% │ ✓ 通过
──────────────────────────────────────────────────────────────

### 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 压测完成，全部通过 |
| 1 | 压测完成，存在超标项 |
| 2 | 采集失败 |
| 3 | 状态检查不通过（未使用 `--force`） |

## 典型工作流

### 场景 1：新机器上架冒烟

```bash
# 1. 确认 stress 已装
which stress || sudo apt install stress

# 2. 两个脚本同目录
ls -l cpu_mem_check.py stress_test.py

```

### 场景 2：交付验收

```bash
# 先看基线状态
python3 cpu_mem_check.py

# 跑标准交付压测
python3 stress_test.py --mode stress --profile 2

# 生成 JSON 报告存档
python3 stress_test.py --mode stress --profile 2 --json > report.json
```

### 场景 3：CI 流水线

```yaml
# .gitlab-ci.yml 示例

  script:
    - python3 stress_test.py --mode stress --profile 1 --json
  allow_failure: true
```

## FAQ

**Q: 为什么 `stress_test.py` 不直接写死核数和内存，而用 `"physical"` / `"auto_80"`？**

A: 因为同一套脚本要跑在不同规格的服务器上。4核8G 和 64核256G 如果都用固定值，要么压不饱，要么压崩。动态解析保证每台机器都压到合理水位。

**Q: `--force` 什么场景用？**

A: 状态检查发现有告警（比如 CPU 已经在 85%），但你需要确认"当前负载 + 压测负载叠加后"的表现，加 `--force` 强制执行。人有最高选择权。

**Q: 压测期间采样间隔能改吗？**

A: 当前硬编码为 3 秒。可以改脚本中 `sample_interval = 3` 这一行，或改为命令行参数覆盖。

**Q: 需要 root 权限吗？**

A: 不需要。`stress` 和脚本采集逻辑都不需要 root。`dmidecode` 是可选增强，有 root 时型号更精确而已。
